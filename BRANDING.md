# BRANDING — Graveyard

## Nome

**Graveyard**

O nome é definitivo. Em inglês por intenção: em cultura tech global, "project graveyard" é um conceito universalmente
compreendido. O produto fala português, o nome fala a língua da indústria — essa tensão é intencional. Não é um
aplicativo de produtividade com nome genérico. É um memorial com nome preciso.

---

## Tagline

> **Projetos mortos. Lições vivas.**

Quatro palavras. Antítese intencional. Captura a filosofia inteira: honrar o que morreu, preservar o que ensinou.

**Variações de uso:**

| Contexto               | Texto                                                               |
| ---------------------- | ------------------------------------------------------------------- |
| Tagline principal      | _Projetos mortos. Lições vivas._                                    |
| Headline expandida     | _Todo projeto que morreu deixou algo vivo em você._                 |
| Eyebrow / label        | _Um memorial digital_                                               |
| Descrição meta / share | _O arquivo para o que foi tentado, não apenas o que foi concluído._ |

---

## Manifesto (versão curta)

Todo desenvolvedor tem uma gaveta de projetos mortos.

Não costumamos falar sobre eles. Os portfólios mostram os sucessos. Os case studies celebram os pivots. Os LinkedIn só
publicam lançamentos. O que ficou para trás — o MVP que ninguém usou, o SaaS com 12 usuários, o app que morreu no
Figma — esse é o trabalho mais honesto que você já fez.

**Graveyard** existe para esse trabalho.

Não para glorificar o fracasso. Não para fingir que abandonar projetos é uma estratégia. Mas para fazer o que
deveríamos ter feito desde o início: documentar com cuidado o que não sobreviveu, entender por que morreu, e preservar
o que deixou vivo.

Projetos merecem epitáfios. Não relatórios.

## Voice & Tone

### Personalidade

| Atributo  | Descrição                                                             |
| --------- | --------------------------------------------------------------------- |
| Elegíaco  | Honra o que foi. Não lamenta, não celebra — reconhece.                |
| Editorial | Escreve como uma revista de qualidade, não como um dashboard de SaaS  |
| Ritual    | Cada interação tem peso. Nada é trivial ou descartável.               |
| Honesto   | Sem eufemismos, sem positividade forçada, sem linguagem de startup    |
| Sóbrio    | Leveza melancólica — não depressivo, não irônico às custas do criador |

### Tom em diferentes contextos

| Contexto           | Tom                       | Exemplo                                                               |
| ------------------ | ------------------------- | --------------------------------------------------------------------- |
| Headings           | Sereno, quase litúrgico   | _"Projetos que quase foram"_                                          |
| Microcopy          | Direto, sem frescura      | _"Nenhum projeto aqui ainda."_                                        |

## Princípios Editoriais

### 1. Projetos não "falham" — eles morrem

Nunca use _"falhou"_, _"não funcionou"_, _"deu errado"_ para descrever um projeto encerrado sem intenção retórica.
Use: **morreu**, **foi sepultado**, **descansou**, **encerrou**, **foi arquivado**.

A distinção não é eufemismo — é filosófica. Um projeto pode morrer por razões completamente alheias ao seu valor.

### 2. Projetos têm nome próprio

Sempre capitalizado. Sempre como se fosse uma pessoa ou uma obra.
_"O Orbit Habit morreu em 2023"_ — não _"o projeto orbit-habit foi descontinuado"_.

### 3. O vocabulário do rito é mantido com consistência

| Termo do rito         | Significado na plataforma                      |
| --------------------- | ---------------------------------------------- |
| **Velório**           | O ato de registrar — abrir a certidão de óbito |
| **Epitáfio**          | A frase definitória do projeto                 |
| **Certidão de óbito** | O formulário de registro completo              |
| **Salão Memorial**    | A galeria de todos os projetos sepultados      |
| **Reencarnação**      | O potencial de retornar em nova forma          |
| **Lições herdadas**   | O que sobreviveu à morte do projeto            |
| **Necrologia**        | O arquivo de dados e estatísticas              |

### 4. Nenhuma linguagem de startup

Evite: _"ship it"_, _"fail fast"_, _"pivot"_, _"learnings"_, _"growth"_, _"traction"_, _"roadmap"_.
Use: _"construiu"_, _"encerrou"_, _"redirecionou"_, _"aprendizados"_, _"crescimento"_, _"rumo"_.

### 5. Peso emocional é dado, não fraqueza

A plataforma reconhece que projetos têm custo afetivo real. Nunca minimize o que o criador sentiu ao construir e ao
abandonar. O peso emocional é um campo de dados tão legítimo quanto a data de morte.

### 6. A ironia tem limite

O conceito tem humor embutido (um velório de software é absurdo e bonito ao mesmo tempo), mas nunca às custas do
projeto ou do criador. O tom é sempre respeitoso com quem criou e com o que foi criado.

## Nomenclatura das Seções

| URL              | Nome no menu      | Descrição                                       |
| ---------------- | ----------------- | ----------------------------------------------- |
| `/`              | —                 | Landing page conceitual                         |
| `/hall`          | Salão Memorial    | A galeria de projetos sepultados                |
| `/hall/[slug]`   | [Nome do projeto] | Obituário individual completo                   |
| `/new`           | Novo Funeral      | Formulário — a certidão de óbito                |
| `/analytics`     | Necrologia        | O arquivo de dados e estatísticas               |
| `/reincarnation` | Reencarnação      | Projetos com potencial de retorno em nova forma |
| `/manifesto`     | Manifesto         | O ensaio fundador                               |

> **Por que "Necrologia"?**  
> Em jornalismo, a necrologia é o registro oficial de mortes — um documento de honra, não de lamento. É mais preciso
> e mais elegante do que "Estatísticas" para um memorial.

## Hierarquia Visual do Copy

```
EYEBROW       → fonte mono, uppercase, tracking wide, gold/50
               ex: "Arquivo · Necrologia"

H1            → fonte serif, bold, parchment
               ex: "O custo do que foi"

SUBTÍTULO     → text-muted-foreground, text-lg, max-w limitado
               ex: "Números são a única honestidade que os projetos deixam para trás."

EPITÁFIO      → fonte serif, italic, gold/70 ou parchment-muted
               ex: "Queria ser lindo demais para funcionar."

LABEL / BADGE → fonte mono, uppercase, tracking, text-2xs
               ex: "PROTOTYPE · Escopo infinito"
```

## Coerência conceito ↔ UI ↔ copy

| Conceito                           | UI                                                   | Copy                                                     |
| ---------------------------------- | ---------------------------------------------------- | -------------------------------------------------------- |
| Morte como transição, não fracasso | Cards dark com borda fina, sem vermelho dominante    | _"morreu"_, _"descansou"_ — nunca _"falhou"_             |
| Ritual e dignidade                 | Ornamentos `✦`, bordas decorativas, tipografia serif | Linguagem litúrgica: _"o ritual"_, _"epitáfio"_          |
| Arquivo vivo                       | Tags, timeline, lições herdadas                      | _"o que sobreviveu"_, _"herdado"_                        |
| Potencial de retorno               | ResurrectionMeter em verdigris (#3d7a6a)             | _"pode voltar"_, _"espera o momento certo"_              |
| Peso emocional real                | Pill com cores semânticas por nível                  | _"assombroso"_, _"nostálgico"_ — não _"score emocional"_ |

## O que nunca fazer

- Usar **"fail"** ou **"falhar"** sem inversão retórica intencional
- CTAs com exclamação: _"Cadastre-se agora!"_ → _"Começar o ritual"_
- Misturar linguagem de growth hacking com o vocabulário do rito
- Usar emojis 🪦 como decoração — os ornamentos tipográficos já bastam
- Chamar a plataforma de _"ferramenta de produtividade"_ ou _"app de gestão"_
- Tratar projetos abandonados como erros — eles foram tentativas legítimas
